Just Fucking Use Cron.

You do not need a distributed workflow orchestration platform with a mascot. You need a schedule and a command. Calm down.

✅ boring ✅ cheap ✅ debuggable ✅ older than your framework
Golden rule: If your requirement is “run this thing every X minutes,” your requirement is not “adopt a whole new religion.” Your requirement is a timer.
Yes, even if you call it “orchestration” to feel important.
Examples Sanity checklist When cron isn’t enough

What cron is great at

  • Daily/weekly/hourly jobs that do one thing and then leave.
  • Cleanup, rollups, exports, backups, “send the report,” “rotate the logs.”
  • “Call this URL every 10 minutes because the business said so.”
  • Giving you a schedule without gifting you a new on-call nightmare.

What you’re probably doing instead (and why it’s dumb)

You built “a scheduler” inside your app.

Now deployments, crashes, and scaling change your schedule. You reinvented cron, but with more bugs and fewer memes.

You added a workflow platform.

Because you needed “run this daily.” You responded by adopting twelve new concepts and a Slack channel.

You run jobs on every instance.

Now your “daily email” sends six times. That’s not “high availability.” That’s “I hate my users.”

You have no alerts.

The job failed three weeks ago and you only noticed when finance asked where the report went. Congrats on your “observability journey.”

Examples

Pick one. Keep it boring. Get back to shipping.

1) Classic crontab (it’s still fine)

# edit your crontab like a civilized person
crontab -e

# run every day at 02:15
15 2 * * * /usr/local/bin/my-report --yesterday >> /var/log/my-report.log 2>&1

# run every 5 minutes
*/5 * * * * /usr/local/bin/ping-healthcheck >> /var/log/ping.log 2>&1

2) systemd timers (cron, but with status + logs)

# systemd timers are great if you want:
# - journal logs
# - easy status checks
# - dependencies
#
# create:
#   /etc/systemd/system/my-job.service
#   /etc/systemd/system/my-job.timer
#
# then:
systemctl enable --now my-job.timer
systemctl status my-job.timer

3) Cron + webhook (yes, you can be “serverless” too)

# trigger an endpoint every hour
0 * * * * curl -fsS https://example.com/jobs/hourly >> /var/log/hourly.log 2>&1
“But we use X cloud scheduler.” Cool. That’s cron with marketing. The point is: stop building a cathedral to run a script.

Sanity checklist (before you invent a platform)

  • Idempotent? Can it run twice without ruining everything? If not: fix that first.
  • Exactly one runner? One host, one “job runner” container, or a leader lock. Not “hope.”
  • Logs? Where do they go, and can a human find them in 30 seconds?
  • Failures? Retry (carefully) or alert. “Silently ignore” is not a strategy.
  • Alerts? If nobody gets notified, it didn’t happen. It’s folklore.
  • Timeouts? Jobs that run forever are just services with commitment issues.

When cron isn’t enough

Rare, but real. Use heavier tools only when you actually need them:

  • Long-running workflows with state machines, human approvals, or multi-step coordination.
  • Massive fan-out where you need distributed backpressure and orchestration.
  • Per-tenant scheduling at huge scale (thousands/millions of independent schedules).
  • Exactly-once across systems (expensive, tricky, and frequently a lie).
Translation: If you can describe your “workflow” as “run this command on a schedule,” you are not building a workflow. You are scheduling a job.

Just fucking use cron.

FAQ (for the professional arguers)

  • “Cron isn’t reliable.” Your infrastructure isn’t reliable. Add monitoring. Also: skill issue.
  • “But retries!” Make it idempotent. Then retry. Or alert. Humans exist.
  • “But locking!” File lock, DB lock, leader election. Choose the simplest one you can explain without a whiteboard.
  • “But we’re serverless!” Cool. Use a scheduler service. That’s still cron. You didn’t transcend physics.
  • “But we need observability!” Great. Start by logging and paging on failure. Not by buying a new dashboard for vibes.
